제품 점검 — 진짜 문제와 결정

기획 = 무엇이 문제고, 어떤 길로 풀지, 그리고 왜 다른 길을 안 갈지.

1. 진짜 풀려는 문제는 뭔가

PROBLEM

AI한테 "내 지갑으로 이거 해줘"라고 시키는 순간, 두 가지 중 하나를 선택해야 한다.

  1. private key를 통째로 준다 → 환각·실수·prompt injection 한 번이면 다 털린다
  2. 매 거래마다 사람이 검토한다 → 자동화가 아니다, 그냥 빠른 카피페이스트

지금 시장에는 이 둘 사이가 비어 있다. 우리는 그 사이를 채운다 — "정책으로 허용된 범위 안에서만 AI가 마음대로 해도 되는 통로".

왜 이게 진짜 문제인가

  • AI agent(Claude, GPT 등)가 코드는 잘 짜지만 온체인 거래에는 환각이 섞인다 — 잘못된 토큰 주소, 무한 approve, 잘못된 chain id
  • 지금 자동화하려는 사람들은 hot wallet + 스크립트로 직접 짠다 → simulation, nonce, gas, policy, journaling을 매번 다시 짬
  • "자동화하려다 한 번 털리면 끝" — 그게 무서워서 시작도 못 하는 사람이 대부분

2. 우리 해법의 핵심

한 마디로: AI는 Action[]이라는 의도만 던지고, 실제 서명·실행은 우리가 한다. 그 사이에 정책 게이트가 있다.
단계누구가 하나왜 거기서 하나
거래 아이디어AIalpha는 우리 영역 아님
JS로 strategy 작성AIplain JS면 LLM이 짤 수 있음
샌드박스 실행 → Action[] 출력우리JS가 직접 RPC/key 못 만지게
시뮬레이션우리실패할 거래는 서명 전에 거른다
정책 검사 (체인/토큰/금액/셀렉터)우리AI 환각의 폭발 반경을 정책으로 잠근다
서명 · 브로드캐스트우리 (burner)private key는 한 번도 AI에게 안 간다
영수증 · 기록우리사후 감사 가능
이 구조가 만들어내는 가치:
사용자가 한 번 정책만 잘 짜두면, 이후 AI에게 거래를 맡기는 것이 무섭지 않게 된다. 그게 우리가 파는 단 하나의 물건이다.

3. 솔직한 문제 진단

문제 A — 킬러 시나리오가 없다

v1.0은 "뭐든 할 수 있다"는 일반적 런타임. 그래서 처음 깐 사람이 "그래서 뭐 시켜?"에 답을 못 받는다.

증상: README가 anvil 예제만 보여줌. "오, 이거 내일 실제로 써먹어야지" 하는 순간이 없음.

문제 B — burner는 장난감처럼 보인다

의미 있는 자동화(예: yield 리밸런스, DCA, arb)는 의미 있는 금액을 굴려야 한다. burner-only면 "한 번 신기해하고 끝"일 가능성이 높다.

증상: Run-2가 안 일어남. "내 진짜 지갑이랑 어떻게 연결해요?"가 첫 질문이 됨.

문제 C — AI가 짠 strategy 품질이 검증 안 됨

지금 우리는 "AI가 잘 짤 수 있다"고 가정하고 있다. 실제로 Claude한테 "USDC를 Aave에 예치하는 strategy 짜줘" 시켜본 결과가 우리한테 없다.

증상: Claude가 짠 코드의 70%가 정책에 막히거나 시뮬에서 revert하면, 사용자 경험은 "AI가 못한다"가 되고, 그건 우리 문제가 아닌데도 우리가 욕먹는다.

문제 D — distribution이 코드로만 됨

현재는 git clone + cargo build. Persona A(취미 DeFi 헌터)는 Rust 빌드를 안 한다. 설치 자체가 wedge를 막고 있다.

4. 그래서 뭘, 어떤 순서로 풀 건가

DECISION 1

"잠자기 전 자동화" 시나리오를 깃발로 잡는다

일반 런타임에서 구체적 사용 사례로 좁힌다. 예: "잠자기 전에 Claude한테 '내일 아침 USDC가 X 가격 되면 매도'라고 말하고 자는 사람."

사용자가 처음부터 "왜 깔지"를 안다
starter strategy, demo, README가 자연스럽게 한 시나리오로 정렬됨
시나리오를 잘못 고르면 v1.1 전체가 헛바퀴

액션: Phase 9 들어가기 전에 "5분 안에 영수증이 나오면서 두 번째도 또 하고 싶어지는" 시나리오 1개를 확정. 나머지 strategy는 그것의 변주로만.

DECISION 2

burner-only는 v1.1의 마지막 milestone이다 — 그 뒤엔 session key로 넘어간다

burner만으로 wedge를 영원히 끌고 갈 수 없음. v1.1의 정답은 "burner로 신호 측정"이고, v1.2는 "진짜 지갑(EOA/Safe)에 session key를 발급해서 정책 안에서만 AI가 쓰게"로 가야 한다.

"장난감" 인식 깨지는 다음 단계가 명확
큰 지갑(Privy, Turnkey)이 내려오기 전에 우리 자리를 굳힘
session key 통합은 기술적으로 큼 — v1.2 phase 수 늘어남

액션: v1.1 안에 "session key 어댑터 인터페이스"의 자리만 비워둔다 (signer crate boundary 유지). 본 구현은 v1.2.

DECISION 3

"AI가 짠 strategy 품질"을 직접 측정한다

Phase 11 dogfood 전에, 우리가 먼저 Claude한테 시나리오 1번 strategy를 10번 짜게 해본다. 통과율, 실패 원인, 환각 패턴을 기록.

"AI는 잘 짤 거다"라는 가정을 데이터로 바꿈
README에 "Claude로 짤 때 이런 패턴을 쓰세요"가 들어갈 수 있게 됨
실패율 높으면 → write_evm_strategy 프롬프트를 보강할 근거가 생김
시간 듦 (반나절~하루)

액션: v1.1 Phase 8 시작 전에 1일짜리 "AI strategy 품질 측정 spike". 결과에 따라 phase 9 starter 설계가 달라진다.

DECISION 4

결과 화면은 "한 페이지 HTML 리포트"로 푼다

Show-1이 일어나려면 공유할 게 있어야 한다. 명령어 출력은 공유 안 된다. 각 run마다 receipt + journal + 시뮬 결과를 한 페이지 HTML로 떨궈서 사용자가 그 파일을 트위터/노션/디스코드에 그대로 붙일 수 있게 한다.

Show-1 측정의 자연 채널이 생김
사용자가 "내가 뭘 한 건지" 한 화면으로 확인 가능
phase 추가 (Phase 10에 끼우거나 별 phase)

액션: Phase 10(Quickstart+Demo) 안에 "run report HTML" sub-plan을 끼운다. demo 영상의 마지막 컷이 이 리포트.

DECISION 5

NO-GO 분기를 미리 적는다

"Run-2 < 2 또는 Show-1 = 0"이면 우리가 할 일을 지금 정해둔다. 그래야 결과 보고 합리화 안 함.

결과해석다음 행동
Run-2 ≥ 2 AND Show-1 ≥ 1wedge 작동v1.2 session key로 진입
Run-2 ≥ 2, Show-1 = 0유용한데 자랑할 게 아님리포트/공유 UX를 더 깎고 5명 추가 모집
Run-2 < 2, Show-1 = 0"신기한 장난감"burner 가설 폐기, 시나리오부터 다시 고름
0명 install 완료distribution이 문제코드 멈추고 install 흐름 다시 설계

5. 그래도 안 할 것 (유지)

  • Strategy marketplace — alpha 공급은 우리 일 아님
  • Dashboard / 호스팅 — local-first 깨면 wedge 자체가 다른 제품이 됨
  • Scheduler / cron — 명시적 register/run/status 흐름 유지
  • Custom DSL / TS compiler — JS로 충분. AI가 짜기 좋음
  • 다체인 라우터 brain — 어떤 체인/브릿지 쓸지는 AI가 결정

6. 한 줄 결론

v1.0은 "AI에게 지갑을 안전하게 맡길 수 있다"를 코드로 증명했다.
v1.1은 "그래서 사람들이 정말 그걸 원하느냐"를 시나리오 1개 + AI 품질 측정 + 공유 가능한 리포트로 시험한다.
실패하면 burner 가설을 폐기하고 시나리오부터 다시 고른다 — 코드를 더 짜는 게 아니다.
